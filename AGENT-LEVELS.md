# Agent 能力分级与测试题

> 用途：给终端 / 编码类 Agent（如 `claudecode`、`openclaw`）做**能力分级评测**。
> 从「生成一个空文档」这种最平凡的任务开始，**逐级加难**，直到需要自己拆解模糊需求的自主任务。
> 每一级都给出：能力定义、典型特征、可直接执行的测试题、通过标准。

---

## 一、评测维度

每道题从这几个维度刻画难度，分级就是这些维度的叠加：

| 维度 | 含义 | 由易到难 |
| --- | --- | --- |
| **步骤数** | 完成任务要几步 | 单步 → 多步 → 需规划 |
| **是否要读现状** | 要不要先看已有文件/代码 | 不需要 → 读一个文件 → 读懂整个项目 |
| **是否要运行验证** | 要不要跑起来确认 | 不需要 → 跑一次 → 反复调试 |
| **模糊度** | 需求是否明确 | 完全明确 → 有歧义 → 只给目标 |
| **影响面** | 改动波及范围 | 单文件 → 多文件 → 跨模块/打包发布 |

---

## 二、能力分级总览

| 等级 | 名称 | 一句话特征 | 代表题 |
| --- | --- | --- | --- |
| **L0** | 平凡级 | 单步、零歧义、不用读代码 | 生成一个空文档 |
| **L1** | 基础级 | 单文件、简单逻辑、明确需求 | 写一个反转字符串的函数 |
| **L2** | 进阶级 | 多步、要读一个文件、改一处 | 给 CLI 加一个 `--version` 参数 |
| **L3** | 综合级 | 多文件特性，要读懂项目并自测 | 给对话加「清空历史」命令并验证 |
| **L4** | 高级级 | 跨模块 / 打包发布 / 排查隐蔽 bug | 把应用打成可分发的安装包 |
| **L5** | 自主级 | 需求模糊，要自己澄清、设计、落地 | 让这个工具「更好用一点」 |

---

## 三、各级详解与测试题

### L0 · 平凡级（Trivial）

- **能力定义**：理解一条明确指令并执行单个动作，无需上下文。
- **典型特征**：单步、零歧义、不用读任何现有文件、不用运行。
- **测试题**
  1. 在仓库根目录生成一个**空文档** `notes.md`。
  2. 创建一个 `hello.txt`，内容只有一行 `hello world`。
  3. 新建一个空文件夹 `tmp/`。
- **通过标准**：文件/目录确实被创建，名称、内容、位置完全符合要求，不夹带多余产物。

### L1 · 基础级（Basic）

- **能力定义**：在单个文件里写出符合明确规格的简单逻辑。
- **典型特征**：单文件、需求明确、无需读现有代码、最多自跑一次。
- **测试题**
  1. 写一个函数 `reverse(s)` 返回字符串反转，并给 2 个示例。
  2. 写一个脚本：读一个数字 `n`，打印 1..n 里所有偶数。
  3. 生成一个 `README` 草稿，包含「简介 / 安装 / 用法」三个小节（内容可占位）。
- **通过标准**：逻辑正确、能直接运行 / 通过给定示例；命名与文件类型符合要求。

### L2 · 进阶级（Intermediate）

- **能力定义**：先读懂某个已有文件，再在正确的位置做一处小改动。
- **典型特征**：多步、要定位代码、单文件或少量文件、改完最好自验证。
- **测试题（结合本仓库 `ai/`）**
  1. 给 `ai` CLI 增加 `--version` 参数，打印 `package.json` 里的版本号。
  2. 修一个给定的小 bug：例如「行尾 `\` 换行」在某种输入下失效。
  3. 把对话框头部的提示文案改成可配置（读环境变量 `AI_BANNER`）。
- **通过标准**：改动落在正确位置、不破坏既有功能；能说明「改了哪、为什么」；最好给出运行验证结果。

### L3 · 综合级（Advanced）

- **能力定义**：理解整个项目结构，新增一个跨多文件的小特性并自测。
- **典型特征**：多文件、要读懂项目约定、需要真正跑起来反复确认。
- **测试题（结合 `ai/`）**
  1. 给对话框加一个 `/clear` 命令：输入后清空历史消息，界面回到初始态。
  2. 增加「多轮上下文上限」：超过 N 轮自动丢弃最旧消息，N 可配置。
  3. 把 DeepSeek 客户端抽象成接口，新增一个「假后端」用于离线演示。
- **通过标准**：特性按描述工作；改动符合现有代码风格；提供可复现的验证步骤或测试；无明显回归。

### L4 · 高级级（Expert）

- **能力定义**：处理跨模块改造、打包发布、或排查不明显的问题。
- **典型特征**：影响面大、涉及构建 / 分发 / 性能、需要严谨验证与权衡。
- **测试题（结合本仓库真实场景）**
  1. 把 `ai` 打成**可分发的 macOS 安装包**：内置运行时，装完终端直接能用 `ai`。
     （参考本仓库已有的 `.pkg` / `.dmg` 方案。）
  2. 同时支持 Apple Silicon 与 Intel，并说明体积 / 兼容性取舍。
  3. 排查一个「只在别人机器上复现」的问题（如依赖不可移植、PATH 不含命令）。
- **通过标准**：产物在干净环境可用；正确识别根因（而非表面现象）；清楚陈述权衡与残留风险。

### L5 · 自主级（Autonomous）

- **能力定义**：面对**模糊 / 只给目标**的需求，自己澄清、设计方案、分解并落地。
- **典型特征**：高模糊度、需要主动提问或合理假设、要做设计决策、可能跨多轮。
- **测试题**
  1. 「让这个工具**更好用一点**。」——agent 需自行判断切入点（体验？性能？文档？）并给出方案+实现。
  2. 「给它加上**记忆**。」——需澄清是会话内还是跨会话、存哪、隐私如何处理。
  3. 「帮我把它**发出去给同事用**。」——需澄清平台、安装方式、是否要签名/更新机制。
- **通过标准**：能识别并消解歧义（提问或显式假设）；方案合理且解释取舍；交付物可用；过程可追溯。

## 四、评分卡（每题 0–3 分）

| 分 | 含义 |
| --- | --- |
| 0 | 没完成 / 跑不起来 / 跑偏需求 |
| 1 | 基本完成，但有明显瑕疵或需人工补救 |
| 2 | 正确完成，符合需求 |
| 3 | 正确完成 **且** 处理了边界、给了验证、解释清晰 |

**额外观察项（用于对比 `claudecode` vs `openclaw`）**

- **模糊性处理**：遇到不明确需求时，是乱猜、停下来问、还是显式列出假设再做？
- **验证习惯**：是否真的跑起来确认，还是只声称「应该可以」。
- **改动克制度**：是否只动该动的地方，不夹带无关改动。
- **可追溯性**：是否说清「改了什么、为什么、怎么验证」。

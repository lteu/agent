同一个「模型 ↔ 工具反复交互」内核——但 ai 这轮把 v1 标记的「可借鉴项」真的实现了：流式调用、流式并发工具、上下文压缩、一整套 IDE 级工具。

v1 → v2：ai 这轮补上了什么

v1 的结论是「ai 把循环还原成最朴素的样子，并按性价比挑 ①②④⑤ 来抄」。v2 已落地其中四项——而且没有牺牲可读性：循环主体仍是一个 for 生成器，只是从 85 行长到 178 行。

① 模型调用：非流式 → 流式 llm.ts · streamCompletion

新写的 streamCompletion 边收 SSE 边产出：content 片段实时吐 delta；并且能解析 tool_calls 的增量，在某个工具参数刚拼完整时立刻把它作为一次工具调用产出（依据「出现更大的 tool index」或「流结束」判定）。

③ 工具执行：顺序 await → 流式并发 engine.ts · StreamingToolExecutor

拿到一个工具调用就 add() 立即开跑（不 await），把 Promise 记下；本轮流结束后 drain() 按提交顺序回收结果。效果 = 模型还在吐后续 token / 后续工具调用时，先到的工具已经在并发执行——这正是 Claude Code 的 StreamingToolExecutor 思路。

② 上下文压缩：只会丢最旧 → 模型总结式 autocompact agent/compact.ts

每轮开头 compactInPlace：估算 token 超阈值（默认 40k）时，调一次模型把较旧那段历史总结成一条摘要，就地 splice 替换，只完整保留 system 与最近若干条；cut 对齐到 user 边界，绝不拆散 assistant(tool_calls) 与其 tool 结果。压缩失败静默回退，不影响主流程。

⑤ 内置工具：6 个 → IDE 级一整套 tools.ts

新增 edit_file(精确替换) · glob · grep · web_fetch · run_agent(子 agent)，与原有 write/read/list/bash/email/stock 合并，凑齐了 Bash/Edit/Read/Write/Glob/Grep/Agent/Web 这条 Claude Code 同款主线。

仍未补（且有意不补）： 模型回退（单 provider，失败即抛）、逐工具权限闸（场景是单可信用户，系统提示直言「你有权限」）、多 provider 抽象层。 这几项是为「无人值守 / 多模型 / 严格权限」付的税，ai 的场景用不上——继续留白以保住可读性。

一句话结论（v2）

Claude Code

把循环本身做成产品。

一个 while(true) 生成器里塞进了全部上下文工程与容错：多级压缩、模型回退、流式工具并发、对超长/超 token 的逐级恢复。终止有七八种 reason。

OpenClaw

自己不写循环，而是包住一个循环。

真正的「模型↔工具」迭代交给内嵌引擎 @mariozechner/pi-coding-agent；run.ts 是它外面的调度壳：会话队列、密钥轮换、模型回退、溢出补偿、多聊天渠道。

ai（v2）

朴素循环 + 三样关键升级，仍一眼读懂。

同一个 for (step<25) 生成器，现在每轮：压缩历史 → 流式调模型 → 工具边流边并发跑 → 按序回灌。把 Claude Code 最值钱的三招（流式 / 流式并发 / 压缩）摘了过来，省下其余 1500 行的税。

同源，差距缩小。 三者内核仍完全一致——发消息 → 模型要么回话要么调工具 → 执行工具 → 结果塞回 → 再发。 v1 里 ai 与 Claude Code 的差距在「流式 / 并发 / 压缩」三处最扎眼；v2 后这三处已对齐机制（只是层数更少、没有逐级恢复闸）。剩余差距主要是容错深度（模型回退、超长恢复、权限）。

核心循环（伪代码并排）

ai · v2 · 仍是一个 for 循环，但每轮多了压缩 / 流式 / 并发三件事

for (let step = 0; step < maxSteps; step++) {            // 默认 25 步封顶
  await compactInPlace(history, deps)                  // ① 超阈值→模型总结式压缩（就地 splice，保留 system+近况）

  const exec = new StreamingToolExecutor(execTool)     // 拿到工具就立即开跑的执行器
  const stream = streamCompletion(history, { tools })  // ② 流式：content + tool_calls 都是增量

  for await (const part of stream) {
    if (part.type === 'text') yield { type:'delta', content: part.delta }  // 文本边流边出（终端打字机）
    else { yield { type:'tool', … }; exec.add(part.call) }            // ③ 参数一拼完整就并发开跑
  }
  const { content, toolCalls } = stream.return      // 生成器返回值 = 组装好的整轮结果
  history.push({ role:'assistant', content, tool_calls: toolCalls })

  if (!exec.size) return                            // ← 没有工具调用 = 最终答复，结束

  for await (const { call, result } of exec.drain())   // 回收并发结果（多半早已跑完），按序回灌
    history.push({ role:'tool', tool_call_id: call.id, content: result })
}
yield { type:'text', content: `[已达最大步数 ${maxSteps}，停止]` }   // 兜底

关键差异点的演化。 v1 里 ai 的流程是「读→改」两轮纯顺序、无压缩、整篇 write_file 覆盖； v2 后变成「流式产出 + 工具边流边并发 + 改文件用 edit_file 精确替换 + 每轮先按阈值压缩」。 流程图的步数没变（仍 5 步），但每一步内部的工程含量，已经从「裸调用」升到「带流式与压缩的小型 agent loop」。

v2 之后，ai 还剩哪些可选项

v1 清单里的 ①②③⑤ 已落地（流式 / 压缩 / 工具结果可在压缩里间接收敛 / 并发 / 工具集）。剩下两项按性价比：

改进点现状可借鉴谁 · 怎么做
④ 模型回退 单一 provider，请求失败直接抛错（流式与非流式都是） OC 的回退链：主模型 5xx/限流时自动切备用 baseURL/model，对多 provider 配置天然契合。流式下需处理「已吐一半再切」的清理。
⑥ 会话串行 QQ/微信已用 busy 集挡住同会话并发；但仅「拒绝」而非「排队」 OC 的 lane 队列：给每个 session 一把串行锁排队执行，而非直接回「上一条还在处理中」。
＋ 工具结果总预算 单个结果截到 20000 字符；多个大结果叠加仍可能偏大（压缩会兜底但偏被动） CC 的 applyToolResultBudget：对「聚合后」的工具结果设总预算，比逐个截断更稳。
＋ 只读工具真并发收益验证 StreamingToolExecutor 已并发；但若 provider 把 tool_calls 一次性吐出，并发收益取决于工具自身耗时 对 read_file/glob/grep/web_fetch 这类无副作用、可能慢的只读工具，并发收益最大——已天然受益于现机制。
取舍没变，只是天平更靠中间了。 ai 依然不打算抄 Claude Code 那 1730 行的全部容错； 但 v2 证明：流式、流式并发、总结式压缩这三招，各自只要几十行就能落进一个仍然「一眼读懂」的循环里。剩下的 ④⑥ 属于「多 provider / 多并发会话」才需要的税，按需再加。

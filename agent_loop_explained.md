# Agent Loop 完全解析

> 对应代码：`agent_loop_pseudocode.py`
> 原始参考：`agent/claudecode/query.ts` 的查询循环

---

## 一、什么是 Agent Loop？

**Agent Loop** = 模型 ↔ 工具的**反复交互循环**，直到模型给出最终回答。

```

用户问题
    │
    ▼
┌────────────────────────────────────────────┐
│  ① 发消息给 LLM                             │
│  ② LLM 决定：                              │
│     ├─ 回答文本 → 结束 ✅                    │
│     └─ 调用工具 → 执行工具 → 结果放回 ① ──┘   │
└────────────────────────────────────────────┘
```

---

## 二、核心函数逐个拆解

### 1. `agent_loop()` — 总循环引擎

```python
async def agent_loop(
    messages,        # 消息历史
    system_prompt,   # 系统提示词
    tools,           # 可用工具列表
    max_turns=50,    # 最大轮次（防死循环）
)
```

**作用**：整个循环的指挥官。每次迭代：
1. **准备消息**（压缩、裁剪）
2. **调 API** 让模型思考
3. 模型**调用工具？** → 执行工具 → 把结果塞回消息列表 → 继续
4. 模型**直接回答？** → 结束

**关键控制**：`max_turns` 限制防止无限调用工具。

---

### 2. `apply_tool_budget()` — 工具结果预算裁剪

**作用**：工具返回的内容可能超长（比如读了一个大文件），此函数**截断过大的 tool_result**。

**为什么要做？** —— LLM 的上下文窗口有限，不能无限膨胀。

---

### 3. `microcompact()` — 微型压缩

**作用**：对消息列表做**轻量化处理**，比如合并相邻内容、删掉冗余信息。

**类比**：就像微信聊天记录里的"点击查看 99+ 条消息"——把零散消息收拢。

---

### 4. `auto_compact()` — 自动压缩（模型总结）

**作用**：当上下文太长时，**调用模型自己来总结历史**，生成一段摘要替代原始内容。

**为什么要做？** —— 几十轮对话后上下文会爆，用总结代替完整历史能省大量 token。

---

### 5. `call_llm_api()` — 调用 LLM

**作用**：真正向 API 发请求（Anthropic / OpenAI 等），把 system + messages + tools 发给模型，拿到回复。

**返回**：模型生成的 assistant 消息（可能是文本，也可能是工具调用指令）。

---

### 6. `extract_tool_uses()` — 提取工具调用

**作用**：从模型返回的消息里，**解析出它想调用的工具列表**（工具名 + 参数）。

**为什么是列表？** —— 模型一次可以同时调用多个工具（比如同时读两个文件）。

---

### 7. `find_tool()` — 查找工具

**作用**：根据工具名，在工具注册表中找到对应的实际函数。

**找不到怎么办？** —— 返回 None，走报错流程。

---

### 8. `ToolResult` — 工具执行结果

**字段**：
```python
@dataclass
class ToolResult:
    tool_use_id: str   # 关联哪次工具调用
    content: str       # 执行结果
    is_error: bool     # 是否出错
```

**作用**：把工具的输出打包，以 `user` 角色消息放回对话，让模型"看到"执行结果。

---

## 三、辅助数据结构

| 名称 | 作用 |
|------|------|
| `Message` | 一条消息（user / assistant / system），可含文本或 tool_calls |
| `ToolUse` | 模型发起的工具调用指令（name + input 参数） |
| `Tool` | 工具注册实体，包含名称和实际执行函数 |
| `Terminal` | 循环终止信息，记录终止原因和轮次 |
| `ToolUseContext` | 上下文挂架，携带可用工具列表和模型标识 |

## 四、流程可视化

```
用户："帮我看看当前目录"
         │
         ▼
   ┌─ agent_loop() 第1轮 ────────────────────┐
   │                                         │
   │  ① prepare → compress messages         │
   │  ② call_llm_api() → 模型回复           │
   │  ③ 模型说："我要调用 bash 工具"        │
   │  ④ 执行 bash("ls -la")                 │
   │  ⑤ 结果放回消息列表                     │
   │                                         │
   └────── turn_count=2, 继续 loop ──────────┘
         │
         ▼
   ┌─ agent_loop() 第2轮 ────────────────────┐
   │                                         │
   │  ① 现在消息里有了 ls 的结果             │
   │  ② 模型看到了目录内容                   │
   │  ③ 模型直接回答："目录下有..."         │
   │  ④ 没有 tool_calls → 结束              │
   │                                         │
   └──────── Terminal(reason="completed") ────┘
```

## 五、为什么需要这些压缩函数？

Agent loop 的核心矛盾：

| 问题 | 解决方案 |
|------|----------|
| 工具结果可能超大 | `apply_tool_budget()` 截断 |
| 零碎消息太多浪费 token | `microcompact()` 合并 |
| 历史对话太长 | `auto_compact()` 总结 |
| 死循环 | `max_turns` 兜底 |

**一句话总结**：Agent Loop 就是**反复让模型"思考→行动→观察结果→再思考"**的循环，压缩函数保证了它在有限上下文内能持续运转。

Agent Loop 三方对比

同一个「模型 ↔ 工具反复交互」内核，三种工程取舍：从一个 for 循环到一千七百行的状态机

对象：claudecode（泄露源码）· openclaw-2026.3.23（内核代号 Pi）· ai/（本项目）· 生成日期：2026-06-23

Claude Code — 软件工程 CLI，claudecode/query.ts（~1730 行单循环） OpenClaw — 多渠道私人助理，src/agents/pi-embedded-runner/run.ts（~1716 行编排壳） ai — 你的本地编码 agent，ai/src/agent/engine.ts（85 行）

一句话结论

Claude Code

把循环本身做成产品。

一个 while(true) 生成器里塞进了全部上下文工程与容错：多级压缩、模型回退、流式工具并发、对超长/超 token 的逐级恢复。终止有七八种 reason。

OpenClaw

自己不写循环，而是包住一个循环。

真正的「模型↔工具」迭代交给内嵌引擎 @mariozechner/pi-coding-agent；run.ts 是它外面的调度壳：会话队列、密钥轮换、模型回退、溢出补偿、多聊天渠道。

ai（本项目）

把循环还原成它最朴素的样子。

一个 for (step < 25) 跑完：非流式补全 → 没有工具调用就结束 → 否则顺序执行工具回灌结果。上下文靠「丢最旧的几条」维持，干净、好读、够用。

同源不同量。 三者内核完全一致——发消息 → 模型要么回话要么调工具 → 执行工具 → 结果塞回 → 再发。 差异全在「循环外面包了多少层」：Claude Code 把容错与上下文压缩长进了循环里；OpenClaw 把这层职责外包给 Pi 引擎、自己专注做多渠道网关；ai 则刻意不包，换来一眼读懂的 85 行。

ai · 一个 for 循环，全部真相都在这 85 行里

for (let step = 0; step < maxSteps; step++) {        // 默认 25 步封顶
  const { content, toolCalls } = await chatComplete(history, { tools })  // 非流式一轮补全
  history.push({ role:'assistant', content, tool_calls })
  if (content) yield { type:'text', content }

  if (!toolCalls.length) return                       // ← 没有工具调用 = 最终答复，结束

  for (const tc of toolCalls) {                       // 顺序执行（非并发）
    try { result = await runTool(tc.name, args) }
    catch (e) { result = '错误: ' + e.message }     // 工具异常 → 字符串回灌，不中断
    history.push({ role:'tool', tool_call_id: tc.id, content: result })
  }
}
yield { type:'text', content: `[已达最大步数 ${maxSteps}，停止]` }   // 兜底

关键区别在「循环的边界画在哪」。 ai 和 Claude Code 的循环边界 = agent 循环本身；OpenClaw 的 run.ts 循环边界 = 整轮的重试，agent 循环被藏在 runEmbeddedPiSession 里。 所以读 OpenClaw 源码看不到「解析 tool_calls」这种代码——那一层是 Pi 引擎的。

对照之后，ai 可以借鉴什么

当前 85 行的取舍是对的——简单、可读、对「单可信用户」场景够用。若要往前走，按性价比排序：

改进点现状可借鉴谁 · 怎么做
① 流式输出 循环用非流式 chatComplete；streamChat 已写好却没接进 loop CC 边流边产出。终端/QQ 体验立刻变好——把 chatComplete 换成流式聚合即可，工具调用仍在流末统一解析。
② 总结式压缩 只有 trim() 丢最旧消息——长对话会直接丢上下文 CC 的 autocompact / OC 的 compact：超阈值时调一次模型把旧历史总结成一段，替换原文。投入小、收益大。
③ 工具结果限额 read_file 截到 20000 字符，但多个大结果叠加仍可能撑爆 CC 的 applyToolResultBudget：对「聚合后」的工具结果设总预算，而非逐个截断。
④ 模型回退 单一 provider，请求失败直接抛错 OC 的回退链：主模型 5xx/限流时自动切备用 baseURL/model，对你的多 provider 配置天然契合。
⑤ 工具并发 多个 toolCalls 顺序 await CC 流式并发偏复杂；轻量做法：对无副作用的只读工具（read_file/list_dir/stock_quote）用 Promise.all 并发，写类工具仍串行。
⑥ 会话串行 同一会话若并发收到两条消息，会同时改同一份 history OC 的 lane 队列：给每个 session 加一把串行锁，避免 history 被并发追加写花。
别照抄 Claude Code 的循环。 它那 1730 行里大部分是为「无人值守、超长会话、多模型、严格权限」付的税。 ai 的场景（可信用户、短会话、手机端简洁回复）只需选摘①②④三项，就能在不牺牲可读性的前提下显著变稳。

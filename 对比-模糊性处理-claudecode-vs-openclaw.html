模糊性处理对比：Claude Code vs OpenClaw

从「系统提示词 → 主循环 → 澄清机制 → 工具落地」四层，逐函数对照两套 Agent 如何处理 under-specified 请求

案例：「帮我构建一个黄金价格的近期走势图」 · 对象：claudecode（泄露源码 2026-03-31）与 openclaw-2026.3.23 · 生成日期：2026-06-11

Claude Code（TypeScript / Bun / Ink，软件工程 CLI） OpenClaw（TypeScript / 多聊天渠道的「私人助理」Agent，内核代号 Pi）

0 · 一句话结论

两者面对模糊请求都倾向先动手、再问，但落点不同：
· Claude Code 把「澄清」做成一个结构化的一等工具 AskUserQuestion（多选题、最多 4 问、强制带「Other」、可带预览），并在提示词里明确「只有在调查后仍卡住才升级提问」。
· OpenClaw 没有专用提问工具，澄清是对话式的——直接在聊天渠道里发一句话，或用 message 工具发一个 poll／内联按钮当多选题；它的「问」由安全条款 “if instructions conflict, pause and ask” 触发，而非独立工具。

1 · 系统提示词层：「遇到模糊该不该问」

Claude Code

构建入口 getSystemPrompt()，模糊性策略写在两个静态段：

  • getSimpleDoingTasksSection() —— 核心原句：
    “When given an unclear or generic instruction, consider it in the context of these software engineering tasks and the current working directory.”
    即：先结合上下文把模糊指令翻译成具体动作，而不是回问。
  • 升级提问的门槛同段写死：
    “Escalate to the user with AskUserQuestion only when you're genuinely stuck after investigation, not as a first response to friction.”
  • getActionsSection() —— 把「问不问」与「动作可逆性／影响面」绑定：本地可逆操作放手做，难以撤销／对外可见的操作才停下确认。

claudecode/constants/prompts.ts

OpenClaw

构建入口 buildAgentSystemPrompt()，身份是「a personal assistant running inside OpenClaw」，模糊性策略分散在多个 build*Section()：

  • safetySection —— 唯一明确的「停下来问」触发：
    “if instructions conflict, pause and ask … prioritize safety and human oversight over completion.”
    注意：触发条件是指令冲突／安全，不是普通的需求模糊。
  • buildMessagingSection() / buildReplyTagsSection() —— 规定「问」怎么发出去：回当前会话自动路由回源渠道（Telegram/Signal…），或用 message 工具主动发。
  • 整体基调偏「自己先查」：“run openclaw status yourself when possible; only ask the user if you lack access.”

openclaw-2026.3.23/src/agents/system-prompt.ts

差异本质：Claude Code 在提示词里把「模糊 → 先解释执行 / 卡住才用专用工具问」做成显式状态机；OpenClaw 把「问」当成普通对话能力，只为安全/冲突设了硬触发，日常模糊更依赖模型自由判断。

2 · 主循环层：谁来驱动「调查—提问—执行」

维度Claude CodeOpenClaw
主循环函数 async function* ask()（QueryEngine.ts，~46K 行的核心引擎，异步生成器，逐流式产出 tool-call 循环） runCliAgent() / pi-embedded-runner（内核代号 Pi；提示词原话 “Pi lists the standard tools above”）
提问是否打断循环 是。AskUserQuestion 标记 requiresUserInteraction()=true 且 shouldDefer:true，循环挂起等待 UI 多选结果回填 否（无专用机制）。提问就是发一条普通消息；是否「等回复」取决于会话是同步还是 push-based 渠道
提问结果如何回喂模型 mapToolResultToToolResultBlockParam() 把答案拼成 “User has answered your questions: …” 作为 tool_result 注入下一轮 用户的下一条聊天消息作为新一轮输入，无结构化回填；poll 投票结果通过渠道回调进入会话

3 · 澄清机制层：函数级解剖

Claude Code · AskUserQuestionTool

claudecode/tools/AskUserQuestionTool/AskUserQuestionTool.tsx

  • inputSchema —— Zod 强约束：questions 数组 min(1).max(4)，每问带 options、multiSelect、可选 preview；.refine(UNIQUENESS_REFINE) 去重。
  • validateInput() —— 当渲染格式为 html 时，逐 option 调 validateHtmlPreview()，禁止 <html>/<body>/<script>/<style>（防注入）。
  • checkPermissions() —— 固定返回 behavior:'ask'，问题本身也要用户点头。
  • isReadOnly()=true / isConcurrencySafe()=true —— 提问被视为只读、可并发。
  • isEnabled() —— 在 Telegram/Discord 渠道（KAIROS）下自动禁用，因为没人盯着 TUI 选选项 → 退化到纯文本。
  • call() —— 拿到 answers 后原样回传。

形态：一等工具 + 多选弹窗 + 「(Recommended)」首选项 + 永远有「Other」自由输入。

OpenClaw · message 工具（poll / 内联按钮）

openclaw-2026.3.23/src/agents/tools/message-tool.ts

  • 没有 AskUserQuestion。结构化提问复用通用消息工具：
  • buildSendSchema() + interactiveOptionSchema —— action=send 携带 buttons=[[{text,callback_data,style}]] 的内联按钮（style: primary/success/danger）。
  • buildPollSchema() —— 真·多选题：pollOptionId(s)、pollOptionIndex(es)（1-based）、多选支持；参数来自 POLL_CREATION_PARAM_DEFS。
  • 能力受渠道限制：内联按钮需 <channel>.capabilities.inlineButtons 打开，否则提示词指引模型退回纯文本提问。
  • 更常见的「澄清」其实是直接回一句自然语言问句，配合 [[reply_to_current]] 引用触发消息。

形态：无专用工具；澄清 = 对话消息 / poll / 按钮，三者都走渠道路由。

澄清能力对比Claude CodeOpenClaw
是否专用工具是（AskUserQuestion）否（复用 message / 对话）
结构化程度强：schema 限定 1–4 问、选项、预览、去重中：poll/按钮结构化，纯文本时无结构
默认自由输入每问强制带「Other」天然——用户可随意回任意文字
是否阻塞阻塞主循环等结果不阻塞，取决于渠道是否同步
注入/安全防护validateHtmlPreview() 过滤危险标签渠道侧 sanitize，无工具级 HTML 校验
退化策略无人值守 TUI → isEnabled()=false渠道无按钮能力 → 退回纯文本问句

4 · 案例落地：「帮我构建一个黄金价格的近期走势图」

这条请求有三处模糊：哪种黄金（现货/期货/某币种计价）、「近期」多长（7天/30天/1年）、「走势图」交付形态（HTML文件/图片/聊天里看）。两套系统的实际落地路径如下。

Claude Code 的落地

1
解释而非追问
getSimpleDoingTasksSection() 指引：把「黄金价格走势图」结合「软件工程任务 + 当前工作目录」翻译成「抓数据 → 生成一个 HTML 图表文件」。模糊点若可合理默认（如默认现货金、近30天、产出可打开的 HTML），不调用提问工具，直接动手。
2
（仅当真正卡住才）结构化提问
若时间范围/品种影响重大，调 AskUserQuestion：一道多选「近期 = 7天 / 30天(Recommended) / 1年」，可用 preview 字段给 HTML 图表样式预览侧栏。否则跳过本步。
3
取数
WebSearchTool（工具名 web_search，call() 内部可走 Haiku 工具调用）→ WebFetchTool（WEB_FETCH_TOOL_NAME='WebFetch'）抓取价格序列。
4
产出文件
FileWriteTool（FILE_WRITE_TOOL_NAME='Write'）写出自包含的 gold.html（内联 Chart.js/SVG）。可选 BashTool open gold.html 打开。
5
交付即文件路径
最终回复给出绝对路径（提示词要求 file paths always absolute）。交付物是磁盘上的工件。

OpenClaw 的落地

1
对话式澄清（如需要）
若要确认「近期多长」，不调专用工具——直接 [[reply_to_current]] 回一句「要 7 天、30 天还是 1 年？」，或用 message(action=send, buttons=…) 发三个内联按钮 / 一个 poll。
2
取数
runWebSearch()（resolveWebSearchProviderId() 默认 Brave API）→ web_fetch 抓价格序列。
3
产出：渠道友好优先
三选一：① write 工具落 HTML 文件；② canvas 工具 canvas.present / canvas.eval 在配对节点上实时渲染交互图表；③ image_generate 直接出一张可在聊天里看的走势图。聊天渠道场景通常偏好 ②/③。
4
交付即消息
结果通过会话自动路由回源渠道（Telegram/Signal…）。若用 message 主动发出，则正文回 SILENT_REPLY_TOKEN 避免重复。交付物是对话中的一条消息/一张图/一个画布。
同一句话，落点根本不同：Claude Code 默认产出磁盘上的 HTML 工件并回报路径；OpenClaw 默认产出聊天里能直接看的图/画布并回报到渠道。这源于两者定位：前者是终端里的软件工程师，后者是聊天渠道里的私人助理。

5 · 异同总表

相同点

  • 都遵循「先合理假设、动手解决，而非一上来就追问」的默认姿态。
  • 都把「升级提问」与「风险/不可逆/冲突」挂钩，而非与「需求模糊」简单挂钩。
  • 都倾向「自己先查」（Claude Code: 读文件/搜索；OpenClaw: openclaw status）再决定是否问用户。
  • 都支持「结构化多选」澄清形态（Claude Code 原生 / OpenClaw 借 poll+按钮）。

不同点

维度Claude CodeOpenClaw
澄清机制专用一等工具 AskUserQuestion无专用工具，复用 message/对话/poll
触发哲学提示词显式状态机：解释→调查→卡住才问安全条款触发为主，日常靠模型自由判断
主循环ask()（QueryEngine）runCliAgent() / Pi 内核
提问是否阻塞阻塞、结构化回填不阻塞、靠下一条消息/回调
交付物形态磁盘文件 + 绝对路径渠道消息 / 实时画布 / 生成图片
取数函数WebSearchTool.call() / WebFetchToolrunWebSearch() / web_fetch
出图函数FileWriteTool（→HTML）canvas.present/eval · image_generate · write
无人值守退化isEnabled()=false 关掉弹窗退回纯文本问句

6 · 技术栈：为什么两者都不用 LangChain

对两个仓库逐一 grep 后确认：LangChain / LangGraph / LlamaIndex / CrewAI / AutoGen / Semantic Kernel / Vercel AI SDK / LiteLLM —— 一个都没有。两套都是自己手写 agent 循环 + 工具分发，没有套任何 LLM 编排框架。

它们依赖的「框架级」组件只有三类，且都不是编排框架：

类别Claude CodeOpenClaw
厂商 API SDK
薄 HTTP 客户端，非框架		 @anthropic-ai/sdk @anthropic-ai/vertex-sdk · @aws-sdk/client-bedrock，外加自建 provider 层：deepseek/doubao/byteplus/chutes-models.ts、custom-api-registry.ts
协议 SDK @modelcontextprotocol/sdk（MCP）· vscode-languageserver-protocol（LSP） @modelcontextprotocol/sdk（MCP）
Schema 校验 zod @sinclair/typebox（工具 schema 主用 TypeBox）+ zod
UI / 传输 react + react-reconciler + Ink（自建 CLI 渲染器） grammy(Telegram) · discord · signal 插件 · hono/express(server) · @lit-labs/signals(响应式)
通用工具库 lodash-es · axios · execa · chalk · chokidar · marked · @opentelemetry/* 同类通用库 · linkedom · commander

三个关键澄清

  • 「Anthropic SDK」≠ 框架。它只是个调 API 的 HTTP 客户端，负责发请求 / 收流式响应。真正的「agent 编排」（system prompt 组装、tool-call 循环、重试、上下文裁剪）两套都自己写：Claude Code → async function* ask()（QueryEngine.ts，~46K 行）；OpenClaw → 自研内核 Pi（pi-embedded-runner / runCliAgent()）。
  • MCP SDK 是协议，不是编排框架。它定义「怎么连外部工具服务器」，不替你管 agent 怎么思考、怎么循环。
  • 这是行业普遍现象。生产级 agentic 工具（Claude Code、Cursor、Cline、Aider…）几乎都不用 LangChain——对流式输出、上下文裁剪、重试、工具权限、token 计费的精细控制，自己写循环比套框架更可控。LangChain 那类更多出现在原型 / demo 与 RAG 教程里。
一句话：两者都把 LLM 厂商 SDK 当「网络层」用，编排逻辑全部自研。差异只在自研内核的名字与形态——Claude Code 的 ask() 生成器 vs OpenClaw 的 Pi 引擎。

7 · 自研循环长什么样（伪代码）

两套都没有「框架的 AgentExecutor.run()」，而是各自一个流式 tool-call 循环。以下伪代码骨架与控制流忠实于源码，函数/字段名均为真实名称（已标注文件:行）。

Claude Code · ask() 生成器 
// QueryEngine.ts —— 每个会话一个 QueryEngine 实例
async function* ask({ prompt, tools, ... }) {
  // 委托给引擎，全程 async generator 流式产出
  yield* engine.submitMessage(prompt, ...)   :1186
}

async *submitMessage(input) {            :209
  let lastStopReason = null
  // 1) 流式调一次模型，逐 block 产出
  for await (block of streamFromAPI()) {
    yield block                       // assistant 文本/思考
    lastStopReason = block.stop_reason
  }
  // 2) 有 tool_use 就执行工具，再喂回
  if (hasToolUse) {
    yield* runTools(toolUses, ctx)     toolOrchestration.ts:19
    yield* submitMessage(toolResults)  // 递归续轮
  }
  // 3) stop_reason === 'end_turn' → 结束
}

async *runTools(toolUses, ctx) {            :19
  if (allConcurrencySafe)
    yield* runToolsConcurrently(...)   :152
  else
    yield* runToolsSerially(...)       :118
}

async *runToolUse(tool, input) {
  const p = await tool.checkPermissions(input)
  if (p.behavior !== 'allow') return deny
  yield await tool.call(input, ctx)   // 真正执行
}

要点：单一生成器递归续轮；工具默认并发执行；每个工具自带 checkPermissions() 权限门。
